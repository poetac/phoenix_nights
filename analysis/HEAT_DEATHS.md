# Updating the heat-death data (hand-verified)

`apps/web/public/data/phx-heat-deaths.json` is the **one dataset on the page that
is not regenerated by a pipeline.** Maricopa County publishes heat-related deaths
in an annual PDF report, not a machine API, so the numbers are transcribed and
hand-verified once a year. This runbook is that process, written down so it is
repeatable and so anyone can audit where every figure came from.

## Source of record

- **Maricopa County Department of Public Health — Heat-Related Deaths Report**
  (one final report per year, usually published the following spring).
- Landing page (always current): <https://www.maricopa.gov/1858/Heat-Surveillance>
- Each year's final report is a PDF under that page's "Heat Reports" archive.

Use the **final** report for a completed year, never the in-season *preliminary*
dashboard — preliminary counts run well below the final confirmed number (deaths
under investigation get added for months after summer ends).

## What the JSON holds

```jsonc
{
  "place": "Maricopa County, AZ",
  "metric": "heat-related deaths (heat-caused + heat-contributed)",
  "source": "...exact report cited...",
  "url": "https://www.maricopa.gov/1858/Heat-Surveillance",
  "series": [ { "year": 2015, "deaths": 84 }, ... ],   // the yearly bar chart
  "demographics": {                                     // the "who the heat finds" grid
    "year": 2024, "total": 608,
    "source": "...report for that year...",
    "note": "...provenance / rounding caveat...",
    "factors": [ { "display": "49%", "label": "...", "sub": "..." }, ... ]
  }
}
```

- `series` drives the deaths-vs-hot-nights chart. It counts **heat-caused +
  heat-contributed** deaths, the county's headline number.
- `demographics` drives the breakdown grid in `HumanCostCard.jsx`. It is for a
  **single** report year (the most recent *complete* demographic report), stated
  in `demographics.year`.

## Yearly update steps

1. Open the new final report PDF from the Heat-Surveillance page.
2. **Series:** append `{ "year": <new year>, "deaths": <final confirmed total> }`.
   Revise the prior year if the county restated it (they sometimes do as late
   cases close).
3. **Demographics:** if the report includes the breakdown, refresh the
   `demographics` block for that year. For each factor transcribe the figure
   **with its denominator** into `sub` (e.g. "of indoor deaths"), and keep the
   framing about systems, not individuals (missing shelter, shut-off power, a
   unit no one could fix). Round as the county rounds; hedge ("more than",
   "nearly") where they do.
4. Update `source` (and `demographics.source`) to cite the exact report and table.
5. `cd apps/web && npm run build` to confirm the JSON parses and the card renders.

## Verification (the "hand-verified" part)

Before committing, confirm **every** number against the primary PDF — not a news
summary. The CI `verify_v0.py` shape-checks the JSON structure but **cannot**
check these values (there is no independent machine source for them), so the
human check is the only guard. Note in the PR which report (year + title) you
read, so the provenance is reviewable.

> If you transcribed any figure from secondary reporting because the PDF was
> unreachable, say so explicitly in the PR and flag it for a primary-source
> re-check before merge — these are sensitive public-health numbers and the page
> holds them to the same "reproduce or reject" bar as everything else.
